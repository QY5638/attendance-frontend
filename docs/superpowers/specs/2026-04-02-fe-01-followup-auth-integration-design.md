# FE-01 Followup 联调增强设计

## 1. 背景

- `FE-01` 已在主分支落地基础能力，包含登录页、公共壳层、角色菜单、受保护路由和基础测试。
- 当前前端已能完成登录与路由守卫，但联调层面的错误提示和鉴权失效回跳仍然偏粗。
- 后端登录接口 `POST /api/auth/login` 已通过集成测试明确以下行为：
  - 登录成功返回 `token`、`roleCode`、`realName`
  - 登录失败仍返回 `HTTP 200`，但 `code = 400`，并区分错误消息
  - 受保护接口未授权返回 `HTTP 401`，消息为 `unauthorized`
  - 已登录但越权访问返回 `HTTP 403`，消息为 `forbidden`

## 2. 目标与非目标

### 2.1 目标

- 将登录失败、字段错误、网络异常、401 会话失效统一归一化为前端可消费的错误对象。
- 让登录页能够根据后端真实错误原因给出更精确的提示，并在存在字段归属时落到具体输入项。
- 统一 401 清理登录态、跳回登录页和重新登录后回原页面的闭环行为。
- 增加 `redirect` 安全校验，避免跳转到非法或外部地址。
- 保持现有 `ADMIN / EMPLOYEE` 菜单、守卫和公共壳层结构不被推翻。

### 2.2 非目标

- 不修改后端接口契约、返回格式或鉴权规则。
- 不引入 refresh token、动态菜单接口、按钮级权限或完整权限中心。
- 不重写现有 FE-01 架构，不做与联调无关的页面重构。
- 不扩展 `FE-02 ~ FE-08` 的业务占位页行为。

## 3. 现状问题

### 3.1 登录错误粒度不足

- `src/api/auth.js` 仅在 `code !== 200` 时直接抛出字符串错误，缺少错误类型与字段归属。
- `src/views/login/LoginView.vue` 只能统一显示 toast，无法将“用户名不能为空”“密码不能为空”等错误准确反馈到表单。

### 3.2 网络错误与业务错误混淆

- 当前请求层没有区分后端业务失败与网络/超时异常。
- 后端未启动、代理失败、请求超时等场景，用户看到的提示会退化为通用“登录失败”。

### 3.3 鉴权失效回跳仍可更严谨

- 401 已能触发清理登录态，但 `redirect` 规则尚未明确约束非法路径。
- 手动退出、会话失效、登录页重复跳转等细节还没有明确定义。

## 4. 设计原则

- 延续现有 `api -> store -> router -> view` 的最小架构，不新增重型认证层。
- 错误归一化集中在 `api/auth.js` 与 `utils/request.js`，页面层只消费统一格式结果。
- `redirect` 只允许受控站内路径，默认宁可回角色首页，也不执行可疑跳转。
- 401 处理以“清状态 + 回登录页 + 合法回跳”为主，不在业务页堆叠重复提示。
- 仅补充与本次联调直接相关的测试，不做无关重构。

## 5. 整体方案

### 5.1 推荐方案

采用“最小闭环增强”方案：

1. 在 `src/api/auth.js` 中归一化登录业务错误。
2. 在 `src/utils/request.js` 中归一化网络错误、超时和 401。
3. 在 `src/store/auth.js` 与 `src/router/index.js` 中收紧登录态失效与回跳逻辑。
4. 在 `src/views/login/LoginView.vue` 中消费归一化错误，改进表单交互。
5. 在单元测试中补齐错误映射、401 行为和 `redirect` 校验。

### 5.2 不采用的方案

- 不将全部登录/权限规则都塞进 `request.js`，避免通用请求层过重。
- 不引入显式登录状态机，避免将 followup 演化成架构重做。

## 6. 错误归一化设计

### 6.1 统一错误结构

前端内部统一使用以下错误对象：

```js
{
  type: 'auth' | 'network' | 'timeout' | 'unauthorized' | 'unknown',
  message: '面向用户展示的中文提示',
  field: '' | 'username' | 'password',
}
```

### 6.2 登录业务错误映射

`src/api/auth.js` 中新增登录错误归一化规则：

- `用户名不能为空` -> `{ type: 'auth', field: 'username', message: '用户名不能为空' }`
- `密码不能为空` -> `{ type: 'auth', field: 'password', message: '密码不能为空' }`
- `用户名或密码错误` -> `{ type: 'auth', field: '', message: '用户名或密码错误' }`
- `账号已禁用` -> `{ type: 'auth', field: '', message: '账号已禁用' }`
- `账号角色已禁用` -> `{ type: 'auth', field: '', message: '账号角色已禁用' }`
- 未知业务错误 -> `{ type: 'auth', field: '', message: 后端 message 或 '登录失败，请稍后重试' }`

### 6.3 网络与超时错误映射

`src/utils/request.js` 中新增通用错误归一化规则：

- 无响应、代理失败、后端未启动 -> `{ type: 'network', message: '无法连接到后端服务，请确认接口服务已启动', field: '' }`
- 超时 -> `{ type: 'timeout', message: '请求超时，请稍后重试', field: '' }`
- 未知异常 -> `{ type: 'unknown', message: '请求失败，请稍后重试', field: '' }`

### 6.4 401 未授权错误

受保护接口返回 `401` 时：

- 前端内部识别为 `{ type: 'unauthorized', message: '登录状态已失效，请重新登录', field: '' }`
- 该错误优先用于驱动“清状态 + 跳登录页”，而不是在当前业务页重复堆叠 toast

## 7. redirect 安全规则

### 7.1 允许的 redirect

只有满足以下条件的值才视为合法 `redirect`：

- 类型为字符串
- 以 `/` 开头
- 不以 `//` 开头
- 不是完整外部地址，如 `http://`、`https://`
- 不等于 `/login`

### 7.2 非法 redirect 处理

以下情况一律视为非法并丢弃：

- 空字符串
- `//example.com`
- `http://example.com`
- `https://example.com`
- `/login`
- 其他异常值

丢弃后回退到当前角色默认首页。

### 7.3 redirect 产生时机

- 未登录访问受保护页：写入当前目标页作为 `redirect`
- 401 发生在受保护页：写入当前目标页作为 `redirect`
- 手动退出：不带 `redirect`
- 当前已在 `/login`：不重复覆盖 `redirect`

### 7.4 登录成功后的跳转优先级

1. 合法 `redirect`
2. 角色默认首页

## 8. 页面交互设计

### 8.1 登录页

- `username` 提交前执行 `trim()`。
- `password` 保持原始值，不做自动裁剪。
- 提交中按钮显示 loading，并禁止重复提交。
- 成功登录后清空旧错误状态。

### 8.2 错误展示

- 当归一化错误带 `field = username` 时，将错误落到用户名输入项。
- 当归一化错误带 `field = password` 时，将错误落到密码输入项。
- 没有字段归属时，显示整表单级错误提示。
- 网络/超时异常时，优先提示用户检查后端服务是否已启动。

### 8.3 401 与退出行为

- 401：清空本地登录态；若当前页是受保护路由且当前路径不是 `/login`，则跳回 `/login` 并带合法 `redirect`；否则直接回 `/login`
- 手动退出：清空状态并跳 `/login`，不保留 `redirect`
- 已在 `/login` 时收到 401：不重复跳转，不重复提示

## 10. 测试设计

### 10.1 auth API 测试

- 业务失败响应被归一化为带 `type / message / field` 的错误对象
- 用户名为空与密码为空可正确识别字段归属

### 10.2 request 测试

- 网络异常被转成固定中文提示
- 超时被转成固定中文提示
- 401 会触发统一未授权处理回调

### 10.3 guard / router 测试

- 非法 `redirect` 被丢弃
- 合法站内 `redirect` 可被保留并回跳
- `/login` 不作为有效 `redirect`

### 10.4 登录页测试

- 提交成功时优先跳合法 `redirect`
- 提交失败时按归一化错误展示
- 提交中阻止重复提交

## 11. 验收标准

- 登录页能够区分常见后端登录失败原因。
- 网络异常时，提示明确区分“服务不可达”与“账号密码错误”。
- 401 会清理登录态并跳回登录页。
- 重新登录后能优先回到合法的原目标页。
- 非法 `redirect` 不会被执行。
- 现有 `ADMIN / EMPLOYEE` 菜单和路由守卫行为不回退。

## 12. 风险与注意事项

- 若后端实际联调返回文案与测试文案不一致，前端错误映射需按后端真实契约微调。
- 登录页如果同时使用表单内错误和全局 toast，需要避免重复提示同一条错误。
- `redirect` 校验必须保持最小、严格，不能为了“方便”放宽为任意字符串跳转。

## 13. 预期产出

- FE-01 登录联调体验更贴近后端实际契约。
- 会话失效回登录与重新登录回原页形成稳定闭环。
- 在不重构 FE-01 架构的前提下，补齐后续业务模块可复用的认证边界能力。
